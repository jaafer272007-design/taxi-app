# خطة مشروع — تطبيق التكسي المشترك بين المحافظات
**اسم العمل:** `[TBD — يُحسم بالـ Phase 0 مع فحص علامة تجارية]`
**النوع:** منصة نقل مشترك (pooled ride-hailing) بين المحافظات، عراقية، Android-first.
**آخر تحديث:** 2026-07-12

---

## 1. تعريف المنتج (مقفل)
تطبيق يربط **ركّاب** يريدون مقعد على مسار بين محافظتين، مع **سواق** يسيّرون هذا المسار — بمشاركة السيارة (seat pooling)، door-to-door، بطلب آني أو مجدول.

**جانبا العرض (كيف تمتلئ السيارة):**
1. **السائق يعلن مسار** (نجف→كربلاء، 4 عصراً، 4 مقاعد) والركّاب يحجزون مقعد. ← **Phase 1**
2. **النظام يجمّع** ركّاب متوافقين (نفس الممر/النافذة) ويلقّيلهم سائق. ← **Phase 2**

**القرار الجوهري:** الوحدة الأساسية بالنظام هي **الرحلة (Trip)** = سيارة تسيّر ممراً بوقت معيّن بعدد مقاعد. بالـ Phase 1 الرحلة يصنعها **السائق**؛ بالـ Phase 2 يصنعها **النظام** أيضاً. نفس نموذج البيانات — بدون إعادة بناء.

**خارج النطاق (مبدئياً — منع تضخّم):** رحلات قصيرة داخل المدينة، دفع رقمي، iOS، مدن متعددة، توصيل طعام/طرود، أي vertical إضافي.

---

## 2. المبادئ الحاكمة
- **Modular monolith أولاً** — لا microservices. نفصل خدمة فقط عند مشكلة scale حقيقية.
- **مرحلة تُثبَت قبل التالية** — ما ننتقل للـ Phase التالي إلا بتحقّق معايير الخروج (exit criteria).
- **Cash-first** — صفر تكامل دفع بالـ MVP (مثل Sehat Beitak).
- **Android-first** — iOS يتأجل للـ Phase 3.
- **WhatsApp OTP** — لا SMS (النت ضعيف وانتشار الواتساب ~80%؛ Careem نفسها اعتمدته بالعراق).
- **ممر واحد أولاً** — النجف↔كربلاء. نثبت السيولة قبل أي توسّع.
- **البيانات تُصمَّم من اليوم الأول لتستوعب الـ pooling** حتى لو ما نبنيه إلا بالـ Phase 2.

## 3. أين نفوز (المشكلة الحقيقية، مو قائمة ميزات)
المنافسون (Obr، Baly، التكسي الوطني الحكومي) موجودون بالبين-محافظات، لكن يفشلون بالضبط بالجزء الصعب:
- **Matching / Liquidity** — Obr: "تحجز ويظل ساعات يدوّر وما يلگّي تكسي". هذا قلب فكرتنا.
- **الثقة والسلامة** — Baly: شكاوى سواق وحوادث. فرصة تمايز كبيرة.
- **عدالة السعر ومعاملة السائق.**

**فلسفة الخطة:** كل مرحلة تُقاس بمقدار ما تحل هالثلاثة، مو بعدد الميزات.

| المهمة | الأهمية | ملاحظة |

## 5. البنية التقنية (توصية — بنود التحقق مُعلّمة ✳)
- **Backend:** Node.js + **NestJS** (modular monolith، modules نظيفة قابلة للفصل لاحقاً).
- **DB:** **PostgreSQL + PostGIS** (استعلامات جغرافية للممرات ونقاط الصعود/النزول).
- **Realtime/Cache:** **Redis** + **WebSocket** (تتبّع مباشر، مواقع السواق).
- **الخرائط والمسارات:** ✳ يُحسم بالـ Phase 0 — خيارات: OSM + OSRM/Valhalla (self-host، أرخص) مقابل Google/Mapbox. تحقق من جودة تغطية OSM لممر النجف↔كربلاء والأسعار الحالية.
- **تطبيقا الموبايل:** **Flutter** — كود واحد للراكب والسائق، Android-first. ✳ تأكد من إصدارات الحزم الحالية عند البدء.
- **Admin:** لوحة ويب (React/Next.js).
- **Auth/OTP:** WhatsApp Business Cloud API. ✳ تأكد من الإعداد الحالي (نفس ما استُخدم بـ Sehat Beitak).
- **Infra:** VPS واحد بالبداية (Docker Compose)؛ managed Postgres إذا متاح.

## 6. نموذج البيانات الأساسي (مُصمَّم للـ Phase 2 من الآن)
> الفكرة: `Trip` كيان موحّد. `createdBy = DRIVER` بالـ Phase 1، ويُضاف `SYSTEM` بالـ Phase 2 بلا تغيير بنيوي.

- **User** — هوية موحّدة، `roles: [RIDER, DRIVER]`.
- **DriverProfile / Vehicle / Document** — بيانات السائق، السيارة، المستمسكات + حالة الاعتماد.
- **Corridor** — ممر باتجاه (Najaf→Karbala / Karbala→Najaf)، نقاط مرجعية.
- **Trip** — `corridorId, driverId (nullable حتى الإسناد), departureTime, seatsTotal, seatsAvailable, pricePerSeat, status, createdBy(DRIVER|SYSTEM)`.
- **SeatBooking** — `tripId, riderId, pickupPoint, dropoffPoint, seatCount, fare, paymentMethod(CASH), status`.
- **SeatRequest** *(Phase 2)* — `riderId, corridor, desiredWindow, pickup, dropoff` → محرّك الـ pooling يستهلكها ليبني `Trip(createdBy=SYSTEM)`.
- **Rating** — `tripId, fromUserId, toUserId, score, comment`.
- **EarningsRecord / CashSettlement** — أرباح السائق (نقدي).
- *(Phase 3)* Wallet، PaymentTransaction، Promo، Surge.

**آلة حالة الرحلة (Trip):** `POSTED → OPEN → FULL/LOCKED → EN_ROUTE → COMPLETED → SETTLED` (+ `CANCELLED`).
**آلة حالة الحجز (SeatBooking):** `REQUESTED → CONFIRMED → ONBOARD → COMPLETED` (+ `CANCELLED / NO_SHOW`).

## 7. المراحل — نظرة عامة

| Phase | العنوان | الهدف الأساسي | معيار الخروج |
|---|---|---|---|
| **0** | التأسيس والتحقق | ترخيص + سواق + سكافولد + ممر محسوم | ريبو جاهز + وضوح تنظيمي + قائمة سواق أولية |
| **1** | MVP: السائق يعلن + حجز مقعد | رحلة نجف↔كربلاء حقيقية من الحجز للنقد | X رحلات/أسبوع + fill rate مقبول على ممر واحد |
| **2** | تجميع النظام + الطلب الآني | النظام يجمّع الركّاب ويسند سائق | نسبة رحلات مُجمَّعة آلياً ناجحة ≥ عتبة |
| **3** | التوسّع والتقوية | مدن/ممرات + دفع رقمي + سلامة + iOS | تشغيل مستقر على ≥3 ممرات |
| **4** | المنصّة (اختياري) | verticals إضافية عند الجدوى | حسب القرار التجاري |

## 8. تفاصيل المراحل

### Phase 0 — التأسيس والتحقق *(بالتوازي مع المسار غير البرمجي)*
**النطاق:**
- حسم الخرائط (OSM/OSRM مقابل Google) + تأكيد تغطية الممر.
- إعداد الريبو، Docker Compose، Postgres/PostGIS/Redis، هيكل NestJS، وملف `CLAUDE.md`.
- تعريف الـ schema الأساسي (قسم 6).
- تصميم شاشات الراكب والسائق (هنا يشتغل سكل `ui-ux-pro-max`).
**الخروج:** بيئة تطوير تعمل + schema مبدئي + وضوح تنظيمي كافٍ للمضي + قائمة سواق أولية للممر.

### Phase 1 — MVP: السائق يعلن مسار + حجز مقعد *(القلب)*
**Backend (modules):**
- `auth` — تسجيل/دخول عبر WhatsApp OTP، أدوار.
- `driver` — ملف السائق، رفع المستمسكات، حالة الاعتماد.
- `trip` — السائق يعلن رحلة (ممر، وقت، مقاعد، سعر/مقعد)، إدارة الحالة.
- `booking` — بحث الراكب بالرحلات المعلنة على الممر + حجز مقعد + نقطة صعود/نزول.
- `trip-lifecycle` — بدء/إنهاء الرحلة، تحديث المقاعد.
- `cash` — تسجيل تحصيل نقدي + أرباح السائق.
- `rating` — تقييم متبادل بعد الرحلة.
- `notification` — إشعارات (حجز جديد، تأكيد، تذكير مغادرة).
**Rider App (Flutter/Android):** onboarding → تصفّح/بحث الرحلات المعلنة نجف↔كربلاء → حجز مقعد → تتبّع → سجل.
**Driver App (Flutter/Android):** onboarding + مستمسكات → إعلان مسار → إدارة المقاعد والحجوزات → بدء/إنهاء → الأرباح.
**Admin Web:** اعتماد السواق + مراجعة المستمسكات، مراقبة الرحلات، إدارة الممرات والتسعير، دعم أساسي.
**آلية الالتزام (ضد no-show):** تأكيد الحجز + سمعة/تقييم. (بلا عربون رقمي بالبداية — cash.)
**الخروج:** رحلات مشتركة حقيقية على الممر، دورة كاملة (حجز → ركوب → نقد → تقييم) تشتغل، مع fill rate مقبول.

### Phase 2 — تجميع النظام (System-Pooling) + الطلب الآني
**النطاق:**
- `seat-request` + **محرّك التجميع**: يجمّع طلبات نفس الممر/النافذة → يبني `Trip(SYSTEM)` ويسند سائق.
- **مجدول-أولاً** (batch assignment) ثم **آني** لاحقاً داخل نفس المرحلة.
- تتبّع مواقع السواق مباشرة (PostGIS + WebSocket) + "أقرب سائق".
- ترتيب نقاط الصعود/النزول (pickup/dropoff sequencing) مع **سقف detour**.
- تسعير بالمقعد أدق (يبقى بلا surge حي).
**الخروج:** نسبة معتبرة من الرحلات مُجمَّعة آلياً بنجاح ورضا ركّاب مقبول.

### Phase 3 — التوسّع والتقوية
ممرات/محافظات إضافية · surge/عروض/كوبونات · دفع رقمي (Zain Cash/Qi Card — الآن يستاهل التكامل) · محفظة · **سلامة**: خيار عائلي/نسائي، زر طوارئ، مشاركة الرحلة · Dispatcher/Control-Room view · Support workspace · **iOS** · تحليلات · فصل الخدمات وين تحتاج فعلاً.

### Phase 4 — المنصّة (اختياري، مستقبلي)
verticals إضافية (توصيل/طرود/نقل مجدول للشركات…) عند وجود جدوى تجارية. هنا تلتقي رؤية الـ 5 تطبيقات/الخدمات المتعددة — **كوجهة نهائية، مو نقطة بداية**.

## 9. المخاطر والتخفيف

| الخطر | التأثير | التخفيف |
|---|---|---|
| **Cold-start / سيولة ضعيفة** | قاتل | ابدأ بـ driver-posts (عرض جاهز) + ممر واحد كثيف + استقطاب سواق موجودين. |
| **بوابة تنظيمية** | يوقف الإطلاق | حسمها بالـ Phase 0 قبل البناء الجدي. |
| **No-show** | يفرّغ السيارات | تأكيد + سمعة الآن؛ عربون رقمي بالـ Phase 3. |
| **حوادث سلامة/ثقة** | سمعة + قانوني | توثيق سواق صارم، تقييم، خيار عائلي/نسائي، تتبّع مشارَك. |
| **النت الضعيف / offline** | تجربة سيئة | حالات offline، إعادة محاولة، تقليل الاعتماد على البث اللحظي بالـ Phase 1. |
| **تكلفة الخرائط** | نزيف تشغيلي | قيّم OSM/OSRM self-host بالـ Phase 0. |
| **بناء نظامي مطابقة سوا** | غرق هندسي | driver-posts بالـ Phase 1، pooling بالـ Phase 2 — لا تبنيهما سوا. |

## 10. مؤشرات النجاح (لكل مرحلة)
- **Phase 1:** عدد الرحلات/أسبوع على الممر · **fill rate** (متوسط المقاعد المباعة/الرحلة) · نسبة إعادة الحجز · نسبة إكمال الرحلة.
- **Phase 2:** نسبة الرحلات المُجمَّعة آلياً · وقت المطابقة · رضا الراكب عن الـ pooling.
- **Phase 3:** ممرات نشطة · معدّل نمو الركّاب/السواق · نسبة الدفع الرقمي.
